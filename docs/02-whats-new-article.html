Context

Why these three, and why we did not just trust ourselves

Prism is a set of command playbooks that split one hard question into a spectrum of agents, judge them, verify the load-bearing claims, and write the result back into project memory. Its own evaluation found something humbling: on open-ended design, a single careful pass beats the full eight-lens fleet, and the fleet only earns its cost when there is a concrete defect to catch.

So we built these three changes the way the evaluation says to. We ran a lean panel for the design, then pointed three cross-tier skeptics at the plan whose only job was to break it. They did. Two of the three designs changed because of what they found. The corrections are called out in each section below, because the corrections are the interesting part.

Architecture

Where the three additions plug in

Addition one

01The Repo Map big codebases shipped

Before this, Prism scoped a codebase with one line of prose: break it into parts, send three to six explorers, scale to size. On a fifty-file project that is fine. On a fifteen-hundred-file monorepo it is guesswork. There was also a promised "file concern map" that routed each lens to the code it owns, but nothing actually built it.

Now /prism-understand runs a size check first. Small repos keep the old flat sweep. Medium and larger repos build a .prism/repo-map.md: the directory tree, the workspace and manifest roots, and a cheap concern tagging, fingerprinted so a later run knows when it is stale.

The skeptic pass cut three things. The first design trusted the cache, used relevance to choose which files to read, and checked staleness by file count. All three were wrong. A trusted cache goes stale without warning. Relevance ranking hides the cross-cutting bug. File count does not move when a function moves between two files of the same size. The final design: the map is a hint, lenses still read their own slice, staleness uses git blob OIDs, and ranking sets read depth, not which files get read. Audits keep a full sweep.

Monorepo build orchestration (a TypeScript front end next to Rust contracts) is detected but not yet fully handled. That is the next slice, deferred on purpose.

Addition two

02The Craft floor clean code shipped

Prism was good at matching a codebase that already exists. It detected the language, the framework, the naming, and the linter, and it conformed. The gap showed up on greenfield work: with nothing to match, there was no standard at all. Code could compile, pass its tests, conform structurally, and still be painful to maintain. Cryptic names, bloated functions, no rationale, dead code left behind.

The Craft floor adds the missing standard. Names reveal intent. A function does one job. No dead code or unused imports survive in the diff. Comments explain why, not what. Leave the file at least as readable as you found it. It sits under the existing "conform first" rule, so it governs only the code that is yours to write.

The skeptic cut the cap and the gate checks. The draft had a thirty-line function cap and proposed heuristic checks in the integrity gate. Both are gone. A line cap breaks long but cohesive units: a Rust match, a JSX tree, a table test. Heuristic checks in a shell script are language-blind and noisy, and a gate that fires false alarms gets ignored, which would erode the secrets-and-faked-tests signal that matters. The floor is qualitative, the gate is untouched, and type discipline stays with the linter that already enforces it.

Addition three

03The /prism-write command human docs shipped

No command wrote documents for a person to read. Prism summarized diffs into its own memory, but it never produced a README, a change summary, or an article. /prism-write fills that gap. It has four modes picked from the target: a grounded README, a change summary, a retroactive comment pass, and a self-contained HTML article with an architecture diagram. The page you are reading is that last mode, run on this very change set.

Two rules carry the weight. Every claim is re-derived from the real files, so a README cannot invent a feature the code lacks. And the voice is human: short, active, concrete, no hype, and no em-dashes. Grounding is the part that actually removes slop. The voice rules only reduce it.

The skeptic caught a contradiction. A style cannot be both a silent default and a forced question. The fix: JetBrains is the default and most modes proceed without asking. Only the article asks, once, and only when there is no brand.md and you have not already chosen. You picked warm monochrome and system fonts, so this page loads with no network requests.

There was also an honest miss worth recording. The command that bans em-dashes first shipped with thirteen of them, inherited from shared boilerplate. A one-line grep caught it, not a careful re-read. The lesson is now in project memory: when a rule is mechanically checkable, run the check on the artifact instead of trusting that you followed it.

How it was built

A lean panel, then three skeptics

The plan came from /prism-plan run in its lean configuration: three explorers to ground the work in real files, then a draft, then three skeptics on different tiers whose only instruction was to refute. They changed the Repo Map and the /prism-write design, and they trimmed the Craft floor. This is the fleet's one measured strength: catching a concrete flaw before it ships. The full plan and every correction live in docs/01-prism-three-improvements.md.

What is next

The production-readiness plan, and what a critic pass cut from it planned

The next body of work is making Prism dependable in real engineering use, planned in docs/03-prism-production-readiness.md. The honest framing is the same one that shapes everything here: Prism is markdown playbooks plus hooks plus memory, not a runtime, so production-grade means dependable artifacts and disciplined process, not uptime.

The first draft of that plan over-built. It proposed a committed symbol and dependency index as the keystone, a ten-file run log, an eight-tier evidence ladder, and a seven-file memory split. A cross-tier critic pass cut almost all of it.

The critics overturned the keystone. A committed index in a markdown repo is stale the moment anyone edits code, which is Prism's own "stale cache is worse than none" lesson. It was replaced with ripgrep plus the Repo Map plus on-demand ctags, never committed. The run log dropped to one checkpoint file for resume, the evidence ladder to four tiers, and the memory split back to one file. The accuracy critic also caught that the draft cited the wrong line numbers in a playbook, which is direct evidence for one of the gaps it found: Prism has no version or drift check, so copied playbooks and line-number citations rot silently.

The lean MVP that survived: decide whether .prism/ memory is team-shared or machine-private (it is gitignored today, so teams get zero shared memory, and the OVERVIEW contradicted the gitignore), add a version and drift check, add a write protocol so two runs do not clobber memory, ship a four-tier evidence ladder with citation enforcement, rewrite the safety guard by reversibility, and add one checkpoint file for resume. Everything heavier waits until an eval fixture proves it earns its cost.

There is also a blunt scorecard in docs/04-path-to-production-grade.md that gives each remaining gap an objective done-bar (planned is not done) and names the ceiling: battle-tested and category-defining are earned by real use and a hardened, re-run eval, not by shipping more features. The fastest credibility jump is to run Prism on real large repos and re-run the eval, because evidence is the one thing you cannot write into a playbook.

Honest limits

What is not done open

  • Monorepo build orchestration is detected but not handled. It is the next slice.
  • The Repo Map saves real work only on repeated runs, and its win is modest. Building a faithful map still costs about one exploration.
  • Anti-slop by instruction is a soft lever. Grounding is the hard one. When a sentence is in doubt, the rule is to cut it.
  • These are playbook changes, now merged to main, but they are instructions, not a script. The real test is running them on a live large repo, which has not happened yet.
